## Validation Layer Architecture

Wave 1.5 explicitly separates two validation layers:

### Layer 1: JSON Schema Validation
- **Purpose:** Syntax and structure validation per JSON Schema Draft 2020-12
- **Engine:** AJV (standards-compliant) or documented custom implementation
- **Scope:** Type checking, required fields, enums, patterns, conditionals
- **Output:** Valid/invalid with schema errors

### Layer 2: OpenMMO Semantic Validation
- **Purpose:** Business logic validation per OpenMMO contracts (T003a, T003b)
- **Engine:** Custom logic (reference resolution, circular detection, target rules)
- **Scope:** ID format semantics, reference existence, circular references, target type validation
- **Output:** Valid/invalid with semantic errors

**Integration:** Layer 1 runs first, Layer 2 runs second. Both must pass for overall validity.

## Workstream A: Validator Hardening Plan

**Dependency:** Workstream B (Gap Audit) must complete first.

### Current State
The Wave 1 validator (`Validator/src/validator.js`) is a custom implementation supporting:
- JSON Schema Draft 2020-12 basics (type, properties, required, enum, pattern)
- Conditional validation (if/then, allOf, anyOf)
- Reference resolution and circular detection
- 28 tests passing

### Recommended Path: Option 1 — Standards-Compliant Engine

**Recommendation:** Migrate to `ajv` (Another JSON Schema Validator) for **Layer 1** only.

**Justification:**
1. **Standards compliance** — AJV is a reference-quality implementation of JSON Schema Draft 2020-12
2. **Ecosystem** — Widely used, actively maintained, extensive test coverage
3. **Performance** — Optimized validation, faster than custom implementation
4. **Maintenance burden** — Eliminates need to maintain custom schema logic
5. **Trust** — External validation of correctness via established library

**Architecture:**
```
Input
  ↓
[Layer 1: AJV] → JSON Schema validation
  ↓ (if valid)
[Layer 2: Custom] → Semantic validation (references, circulars)
  ↓
Output
```

### Migration Plan

| Phase | Action | Risk | Mitigation |
|-------|--------|------|------------|
| 1 | Complete Gap Audit (B1-B5) | — | Prerequisite |
| 2 | Add AJV dependency | Low | Add to package.json |
| 3 | Create AJV wrapper for Layer 1 | Medium | Maintain same public API |
| 4 | Port schema loading | Low | File loading unchanged |
| 5 | Extract Layer 2 (semantic) | Medium | Preserve custom logic for T003b |
| 6 | Port circular detection | Medium | Preserve algorithm |
| 7 | Run existing tests | Low | All 28 must pass |
| 8 | Add AJV-specific tests | Low | Test engine behavior |

### Schema/Fixture Change Policy

**Explicit Policy:**
- ✅ **Allowed:** Corrections to schemas/fixtures where required to fix truth mismatches or proven defects
- ✅ **Allowed:** Documentation updates to align with actual behavior
- ❌ **Not Allowed:** New schema surface area (new types, fields, enums)
- ❌ **Not Allowed:** New fixtures for new features

**Decision Gate:** Any schema/fixture change requires:
1. Gap audit evidence showing truth mismatch or defect
2. QA Validation Lead approval
3. Spec Steward approval (for schema changes)
4. Chief Architect approval (for structural changes)

### Compatibility Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| AJV behavior differs from custom | Medium | High | Comprehensive test comparison |
| Schema features unsupported | Low | Medium | Verify all Wave 1 features work |
| Error message format changes | High | Low | Update CLI to normalize messages |
| Performance regression | Low | Low | Benchmark before/after |
| Layer separation complexity | Medium | Medium | Clear API boundaries |

### Success Criteria
- Gap audit B1-B5 complete before migration starts
- All 28 existing tests pass without modification
- Error messages remain human-readable
- Validation performance improves or stays equivalent
- Layer 1 (AJV) and Layer 2 (semantic) clearly separated

### Alternative: Option 2 (Fallback)
If AJV migration proves infeasible:
- Document custom validator behavior explicitly
- Add comprehensive edge-case tests

- Document deviations from standard
- Maintain Layer 1 / Layer 2 separation conceptually

## Workstream D: Contract-Test / Golden-Output Plan

### Golden-Output Scope Definition

**Machine-Readable Output (Strongly Versioned):**
- **Format:** JSON output (`--json` flag)
- **Stability:** Schema versioned in output
- **Testing:** Exact match required in contract tests
- **Versioning:** Major version bumps on breaking changes
- **Example:**
```json
{
  "version": "1.0.0",
  "valid": false,
  "errors": [
    {
      "type": "REQUIRED_FIELD_MISSING",
      "path": "name",
      "message": "Required field missing: name"
    }
  ]
}
```

**Human-Readable Output (Normalized/Practical):**
- **Format:** Terminal output with colors/symbols
- **Stability:** Structure preserved, formatting may evolve
- **Testing:** Structure match (✓/✗, error presence), not exact string match
- **Versioning:** Not versioned, but changes reviewed
- **Example:**
```
✗ file.json
  [REQUIRED_FIELD_MISSING] at name Required field missing: name
✗ Validation failed: 1 error(s)
```

### Reference Contract Tests

**Valid References:**
- Item → Reward reference exists
- Reward → Item reference exists

**Circular References:**
- Item A → Reward X → Item A (circular)
- Item A → Reward X → Item B (not circular)

### CLI Output Contracts

**Machine-Readable (Exact Match Required):**
```json
{
  "version": "1.0.0",
  "valid": true,
  "errors": [],
  "data": {...}
}
```

**Human-Readable (Structure Match Required):**
- Presence of ✓ or ✗ indicator
- Presence of error type and path
- Presence of summary line

**Exit Code Contract:**
| Scenario | Expected Exit Code |
|----------|-------------------|
| All valid | 0 |
| Validation errors | 1 |
| Invalid args | 2 |
| Internal error | 3 |

### Contract Test Runner

```javascript
// Contracts/run-contracts.js
const contracts = loadContracts();
let passed = 0;
let failed = 0;

for (const contract of contracts) {
  const result = runContract(contract);
  if (matchesExpected(result, contract.expected)) {
    passed++;
  } else {
    failed++;
    console.error(`Contract ${contract.id} failed`);
  }
}

process.exit(failed > 0 ? 1 : 0);
```

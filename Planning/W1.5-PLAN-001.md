## Workstream A: Validator Hardening Plan

### Current State
The Wave 1 validator (`Validator/src/validator.js`) is a custom implementation supporting:
- JSON Schema Draft 2020-12 basics (type, properties, required, enum, pattern)
- Conditional validation (if/then, allOf, anyOf)
- Reference resolution and circular detection
- 28 tests passing

### Recommended Path: Option 1 — Standards-Compliant Engine

**Recommendation:** Migrate to `ajv` (Another JSON Schema Validator)

**Justification:**
1. **Standards compliance** — AJV is a reference-quality implementation of JSON Schema Draft 2020-12
2. **Ecosystem** — Widely used, actively maintained, extensive test coverage
3. **Performance** — Optimized validation, faster than custom implementation
4. **Maintenance burden** — Eliminates need to maintain custom schema logic
5. **Trust** — External validation of correctness via established library

### Migration Plan

| Phase | Action | Risk | Mitigation |
|-------|--------|------|------------|
| 1 | Add AJV dependency | Low | Add to package.json |
| 2 | Create AJV wrapper | Medium | Maintain same public API |
| 3 | Port schema loading | Low | File loading unchanged |
| 4 | Port reference validation | Medium | Custom logic for T003b |
| 5 | Port circular detection | Medium | Preserve algorithm |
| 6 | Run existing tests | Low | All 28 must pass |
| 7 | Add AJV-specific tests | Low | Test engine behavior |

### Compatibility Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| AJV behavior differs from custom | Medium | High | Comprehensive test comparison |
| Schema features unsupported | Low | Medium | Verify all Wave 1 features work |
| Error message format changes | High | Low | Update CLI to normalize messages |
| Performance regression | Low | Low | Benchmark before/after |

### Success Criteria
- All 28 existing tests pass without modification
- Error messages remain human-readable
- Validation performance improves or stays equivalent
- No changes required to schemas or fixtures

### Alternative: Option 2 (Fallback)
If AJV migration proves infeasible:
- Document custom validator behavior explicitly
- Add comprehensive edge-case tests
- Add JSON Schema compliance test suite
- Document deviations from standard

### Gap B1: Recursive Discovery

**Issue:** `validateProject()` only checks `Fixtures/items/` and `Fixtures/rewards/` hardcoded.

**Fix:**
```javascript
// Support recursive discovery
function discoverFiles(dir, pattern = '*.json') {
  // Recursively find all JSON files
  // Return array of paths
}
```

**Test:**
- Create nested fixture directories
- Verify discovery finds all files

### Gap B2: Reference Validation Integration

**Issue:** Reference validation requires explicit call to `validateReferences()`.

**Fix:**
```javascript
// Integrate into validateFile()
function validateFile(filePath, schemaName, options = {}) {
  const result = validateSchema(data, schema);
  
  if (options.checkReferences !== false) {
    const refErrors = validateReferences(data, ...);
    result.errors.push(...refErrors);
  }
  
  return result;
}
```

**Test:**
- Fixture with invalid reference should fail by default
- Option to skip reference checks when needed

### Gap B3: Conditional Validation

**Issue:** if/then validation has limited test coverage.

**Fix:**
- Add tests for equipment_slot requirement
- Add tests for stackable conditional
- Add tests for complex nested conditionals

### Gap B4: CLI/Validator Alignment

**Issue:** Potential drift between CLI and validator behavior.

**Fix:**
- Audit CLI code for any validation logic duplication
- Ensure CLI uses validator API exclusively
- Add integration tests for CLI + validator

### Success Criteria
- All gaps resolved
- New tests added for each gap
- Existing tests still pass

## Workstream C: CI and Automation Plan

### Pipeline Scope

```yaml
# .github/workflows/wave1-ci.yml
name: Wave 1 CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  validator-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: cd Validator && npm test
      
  cli-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: cd CLI && npm test
      
  fixture-validation:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: ./CLI/bin/openmmo validate --all
      
  docs-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          # Check for broken internal links
          grep -r '\[.*\](.*\.md)' docs/ README.md | while read line; do
            # Extract and verify links
          done
```

### Triggers
- **Push to main:** Full pipeline
- **Pull request:** Full pipeline
- **Push to feature branch:** Validator + CLI tests only

### Required Pass Conditions
- Validator tests: 100% pass
- CLI tests: 100% pass
- Fixture validation: 100% valid fixtures pass, invalid fixtures fail as expected
- Docs check: No broken internal links

### Blocking Behavior
- CI failures block PR merges (branch protection)
- CI failures block Wave 2 planning activation
- Weekly CI health reports to #openmmo-director

## Workstream D: Contract-Test / Golden-Output Plan

### Contract Test Pack Structure

```
Contracts/
├── schema/
│   ├── valid-cases.json
│   └── invalid-cases.json
├── reference/
│   ├── valid-references.json
│   └── circular-references.json
├── cli/
│   ├── human-output.txt
│   └── json-output.json
└── exit-codes/
    └── exit-code-cases.json
```

### Schema Contract Tests

**Valid Cases:**
```json
{
  "contract_id": "SCHEMA-001",
  "description": "Minimal valid item",
  "input": {
    "id": "item_test",
    "name": "Test",
    "item_type": "material"
  },
  "expected": {
    "valid": true,
    "errors": []
  }
}
```

**Invalid Cases:**
```json
{
  "contract_id": "SCHEMA-002",
  "description": "Missing required field",
  "input": {
    "id": "item_test"
  },
  "expected": {
    "valid": false,
    "errors": [
      {
        "type": "REQUIRED_FIELD_MISSING",
        "path": "name"
      }
    ]
  }
}
```

### Reference Contract Tests

**Valid References:**
- Item → Reward reference exists
- Reward → Item reference exists

**Circular References:**
- Item A → Reward X → Item A (circular)
- Item A → Reward X → Item B (not circular)

### CLI Output Contracts

**Human-Readable Output (Golden):**
```
✓ Fixtures/items/valid/01_consumable_potion.json
✓ Validation passed
```

**JSON Output Contract:**
```json
{
  "valid": true,
  "errors": [],
  "data": {...}
}
```

**Exit Code Contract:**
| Scenario | Expected Exit Code |
|----------|-------------------|
| All valid | 0 |
| Validation errors | 1 |
| Invalid args | 2 |
| Internal error | 3 |

### Contract Test Runner

```javascript
// contracts/run-contracts.js
const contracts = loadContracts();
let passed = 0;
let failed = 0;

for (const contract of contracts) {
  const result = runContract(contract);
  if (matchesExpected(result, contract.expected)) {
    passed++;
  } else {
    failed++;
    console.error(`Contract ${contract.id} failed`);
  }
}

process.exit(failed > 0 ? 1 : 0);
```

### Success Criteria
- Contract test pack covers all major behaviors
- Contract tests run in CI
- Contract failures block PRs
- Golden outputs version-controlled
